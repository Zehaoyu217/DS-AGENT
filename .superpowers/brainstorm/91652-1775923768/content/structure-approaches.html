Project Structure: 3 Approaches

How to organize claude-code-agent as a full-stack analytical platform

A

Layered Monorepo

Top-level directories by technical layer. Clear separation of concerns. Each layer is independent.

claude-code-agent/
├── reference/              # src/ renamed — read-only CLI source
│   └── src/                # original 512K TS source
├── backend/                # Python/FastAPI — YOUR code
│   ├── agent/              # agent loop, LangGraph, tools
│   ├── sandbox/            # Python execution sandbox
│   ├── data/               # ingestion, DuckDB, datasets
│   ├── api/                # REST + SSE endpoints
│   ├── models/             # Pydantic models, schemas
│   └── config/             # settings, env management
├── frontend/               # React+Vite analytical UI
│   └── src/
│       ├── components/     # UI components
│       ├── panels/         # 4-zone layout panels
│       ├── devtools/       # developer mode UI
│       └── stores/         # state management
├── mcp/                    # MCP server (existing)
├── infra/                  # docker, helm, grafana, ollama
├── wiki/                   # LLM wiki (Karpathy pattern)
├── docs/                   # harness: architecture, ADRs, SOPs
├── tools/                  # graphify, scripts, dev utilities
├── tests/                  # cross-cutting test suites
└── CLAUDE.md               # entry point
✓ Obvious where everything lives
✓ Easy to navigate for newcomers
✗ Backend/frontend coupling hidden
✗ Shared types need a convention
B

Domain-Driven

Organized by business domain. Features grouped by what they DO, not what layer they're in.

claude-code-agent/
├── platform/               # the analytical product
│   ├── agent/              # agent loop + tools
│   ├── sandbox/            # execution environment
│   ├── data/               # ingestion + DuckDB
│   ├── api/                # REST/SSE layer
│   └── ui/                 # React+Vite frontend
├── devtools/               # developer mode
│   ├── dashboard/          # ops + observability UI
│   ├── control/            # config management
│   └── cli/                # dev CLI scripts
├── knowledge/              # wiki + graphify + docs
│   ├── wiki/               # LLM wiki
│   ├── graphs/             # graphify output
│   └── docs/               # architecture, ADRs, SOPs
├── reference/              # CLI source (read-only)
├── infra/                  # docker, helm, grafana
└── CLAUDE.md
✓ Features are co-located
✓ Devtools is first-class citizen
✗ Deeper nesting
✗ "platform" is vague as a name
C

Hybrid (Recommended)

Layer separation at top level, domain organization within layers. Best of both worlds.

claude-code-agent/
├── backend/                # Python/FastAPI
│   ├── app/
│   │   ├── agent/          # LangGraph agent, tools, prompts
│   │   ├── sandbox/        # Python code execution
│   │   ├── data/           # ingestion, DuckDB, datasets
│   │   ├── api/            # endpoints (REST + SSE)
│   │   ├── wiki/           # LLM wiki engine
│   │   └── config/         # settings, model registry
│   ├── tests/
│   └── pyproject.toml
├── frontend/               # React+Vite
│   └── src/
│       ├── panels/         # chat, artifacts, trace, left
│       ├── devtools/       # developer mode (ops + config)
│       ├── components/     # shared UI
│       └── stores/         # zustand/jotai state
├── reference/              # read-only CLI source
│   └── src/
├── mcp/                    # MCP server
├── infra/                  # docker, helm, grafana, ollama
├── knowledge/              # project intelligence
│   ├── wiki/               # LLM wiki pages
│   ├── graphs/             # graphify output
│   └── adr/                # architecture decisions
├── docs/                   # SOPs, guides, gotchas
│   ├── architecture.md
│   ├── testing.md
│   ├── deployment.md
│   └── gotchas.md
├── scripts/                # dev tooling
├── CLAUDE.md               # harness entry point
└── docker-compose.yml
✓ Clear layers at top level
✓ Domain organization within each layer
✓ Knowledge is first-class (wiki + graphs + ADRs)
✓ Devtools embedded in frontend (where it renders)
✓ Backend/frontend independently deployable
~ Wiki engine in backend, wiki content in knowledge/
